## 宿題を始める前に：重要な概念の復習

### Gitとは何か

Git（ギット）は、ソースコードのバージョン管理システムです。2005年にLinuxの開発者であるリーナス・トーバルズによって開発されました。

**バージョン管理とは：**
ファイルの変更履歴を記録し、過去の任意の時点に戻れるようにすることです。Word文書で「レポート_v1.docx」「レポート_v2.docx」「レポート_最終版.docx」「レポート_最終版_修正.docx」のようにファイル名を変えて管理したことはありませんか？Gitを使えば、このような管理を自動的に、より効率的に行えます。

**Gitの基本用語：**

- **リポジトリ（Repository）**: プロジェクトのファイルと変更履歴を保存する場所。「リポ」「レポ」と略されることもあります。
- **コミット（Commit）**: 変更を記録すること。「保存」に近いイメージですが、単なる保存ではなく「この時点の状態を記録する」という意味合いがあります。
- **ステージング（Staging）**: コミットする前に、どのファイルの変更を記録するか選ぶこと。
- **ブランチ（Branch）**: 開発の流れを分岐させること。メインの開発とは別に、新機能を試したり、バグを修正したりできます。
- **マージ（Merge）**: 分岐したブランチを統合すること。
- **プッシュ（Push）**: ローカル（自分のパソコン）の変更を、リモート（GitHubなど）に送信すること。
- **プル（Pull）**: リモートの変更を、ローカルに取り込むこと。

### GitHubとは何か

GitHub（ギットハブ）は、Gitリポジトリをホスティング（保管）するWebサービスです。Microsoftが運営しています。

GitHubの主な機能：
- **リポジトリのホスティング**: コードをクラウド上に保存
- **コラボレーション**: 複数人での共同開発
- **Issue**: バグや機能要望の管理
- **Pull Request**: コードレビューと変更のマージ
- **Actions**: CI/CD（自動テスト・自動デプロイ）
- **Pages**: 静的サイトのホスティング

世界中の開発者がGitHubを使っており、オープンソースプロジェクトの多くがGitHub上で公開されています。

### デプロイとは何か

デプロイ（Deploy）とは、開発したアプリケーションを、実際にユーザーがアクセスできる環境に配置することです。

日本語では「本番環境への展開」「公開」などと呼ばれます。

開発中は、自分のパソコン上でアプリを動かしています（`localhost:3000` など）。この状態では、他の人はアプリにアクセスできません。デプロイすることで、インターネット経由で誰でもアクセスできるようになります。

### Vercelとは何か

Vercel（ヴァーセル）は、Next.jsを開発した会社が提供するクラウドプラットフォームです。Webアプリケーションのデプロイが非常に簡単にできます。

Vercelの特徴：
- **無料枠が充実**: 個人利用であれば、ほとんどの機能が無料
- **GitHubとの連携**: リポジトリをプッシュするだけで自動デプロイ
- **HTTPS対応**: セキュアな通信が自動で設定される
- **CDN**: 世界中のサーバーからコンテンツを配信
- **プレビュー環境**: ブランチごとに自動でプレビューURLを生成

**CDN（Content Delivery Network）とは：**
世界中に分散配置されたサーバー群のことです。ユーザーに最も近いサーバーからコンテンツを配信することで、読み込み速度を高速化します。

**HTTPS とは：**
HTTP（Webの通信プロトコル）に暗号化を追加したものです。URLが `https://` で始まるサイトは、通信が暗号化されているため、第三者に内容を盗み見られるリスクが低くなります。

## Step 1: GitHubでリポジトリを作成する（所要時間：約10分）

### リポジトリとは（復習）

リポジトリは、プロジェクトのすべてのファイルと、その変更履歴を保存する場所です。「フォルダ」に似ていますが、単なるフォルダではなく、いつ・誰が・何を変更したかの記録も含まれています。

### 手順

**1-1: GitHubにアクセスする**

1. ブラウザで `github.com` にアクセス
2. 右上の「Sign in」からログイン

**1-2: 新しいリポジトリを作成する**

1. 右上の「+」ボタンをクリック
2. 「New repository」を選択
3. 以下の情報を入力：
   - **Repository name**: アプリの名前（例：`my-todo-app`）
     - 半角英数字とハイフンのみ使用可能
     - スペースは使えません
   - **Description**: アプリの簡単な説明（任意）
   - **Public / Private**: 「Public」を選択
     - Public: 誰でもコードを見られる
     - Private: 自分だけがコードを見られる
   - **Initialize this repository with**: 何もチェックしない
4. 「Create repository」をクリック

**1-3: 作成完了画面を確認する**

リポジトリが作成されると、「Quick setup」という画面が表示されます。この画面に表示されているコマンドを、次のステップで使います。

特に以下の2つの情報をメモしておいてください：
- リポジトリのURL（`https://github.com/あなたのユーザー名/リポジトリ名.git`）
- 「…or push an existing repository from the command line」のセクションに書かれているコマンド

### この段階での提出物

GitHubのリポジトリ作成完了画面のスクリーンショット

## Step 2: AntigravityからGitHubにプッシュする（所要時間：約20分）

### 手順

**2-1: Antigravityでターミナルを開く**

1. Antigravityでプロジェクトを開く
2. 画面下部の「Terminal」タブをクリック

**2-2: Gitの初期設定をする（初めての場合のみ）**

Gitを初めて使う場合は、ユーザー名とメールアドレスを設定する必要があります。

以下のコマンドを実行してください（自分の情報に置き換えて）：

```bash
git config --global user.name "あなたの名前"
git config --global user.email "あなたのメールアドレス"
```

例：
```bash
git config --global user.name "Taro Tanaka"
git config --global user.email "taro@example.com"
```

この設定は一度行えば、以後は不要です。

**2-3: リポジトリを初期化する**

プロジェクトがまだGitリポジトリになっていない場合は、初期化が必要です。

```bash
git init
```

**git init とは：**
現在のフォルダをGitリポジトリとして初期化するコマンドです。これにより、`.git` という隠しフォルダが作成され、バージョン管理が始まります。

**2-4: 変更をステージングする**

すべてのファイルをステージングします。

```bash
git add .
```

**git add とは：**
コミット対象のファイルを「ステージングエリア」に追加するコマンドです。`.`（ドット）は「現在のフォルダ内のすべてのファイル」を意味します。

**2-5: 変更をコミットする**

```bash
git commit -m "Initial commit: アプリの初期バージョン"
```

**git commit とは：**
ステージングしたファイルの変更を記録するコマンドです。`-m` オプションの後に、この変更の説明（コミットメッセージ）を記述します。

コミットメッセージは、後から変更履歴を見たときに「何をしたか」がわかるように書きます。良いコミットメッセージの例：
- 「ログイン機能を追加」
- 「商品一覧のバグを修正」
- 「デザインを改善」

**2-6: リモートリポジトリを設定する**

GitHub上のリポジトリを「リモート」として設定します。

```bash
git remote add origin https://github.com/あなたのユーザー名/リポジトリ名.git
```

**git remote add とは：**
リモートリポジトリを登録するコマンドです。`origin` は慣例的に使われる名前で、メインのリモートリポジトリを指します。

**2-7: プッシュする**

ローカルの変更をGitHubに送信します。

```bash
git branch -M main
git push -u origin main
```

**各コマンドの説明：**
- `git branch -M main`: 現在のブランチ名を `main` に変更（GitHubのデフォルトブランチ名に合わせる）
- `git push -u origin main`: `origin`（GitHub）の `main` ブランチに変更をプッシュ。`-u` は「以後、このブランチのプッシュ先をorigin/mainに設定する」という意味

初回のプッシュ時には、GitHubの認証を求められる場合があります。指示に従って認証してください。

**2-8: GitHubで確認する**

ブラウザでGitHubのリポジトリページをリロードして、ファイルがアップロードされていることを確認してください。

### この段階での提出物

- GitHubリポジトリのページ（ファイルが表示されている状態）のスクリーンショット
- または、リポジトリのURL

## Step 3: Vercelでデプロイする（所要時間：約20分）

### 手順

**3-1: Vercelにアクセスする**

1. ブラウザで `vercel.com` にアクセス
2. 「Log in」をクリック
3. 「Continue with GitHub」を選択してログイン

**3-2: 新しいプロジェクトを作成する**

1. ダッシュボードで「Add New...」→「Project」をクリック
2. 「Import Git Repository」セクションで、先ほど作成したリポジトリを探す
3. 見つかったら「Import」をクリック

**3-3: プロジェクト設定を確認する**

以下の設定を確認・調整します：

- **Project Name**: デプロイ後のURLに使われます。変更しても構いません。
- **Framework Preset**: Next.jsが自動検出されていることを確認
- **Root Directory**: 通常は変更不要
- **Build and Output Settings**: 通常は変更不要

**3-4: 環境変数を設定する（重要）**

Antigravityの `.env.local` ファイルに設定した環境変数を、Vercelにも設定する必要があります。

1. 「Environment Variables」セクションを展開
2. 以下の変数を追加：
   - **Name**: `NEXT_PUBLIC_SUPABASE_URL`
   - **Value**: `.env.local` に記載されている値をコピー
3. 「Add」をクリック
4. 同様に、`NEXT_PUBLIC_SUPABASE_ANON_KEY` も追加
5. Gemini APIを使っている場合は、`GEMINI_API_KEY` なども追加

環境変数の設定は非常に重要です。これを忘れると、デプロイ後にアプリが正常に動作しません。

**3-5: デプロイを実行する**

1. 「Deploy」ボタンをクリック
2. ビルドが始まります（1〜3分程度）
3. 「Congratulations!」と表示されれば成功

**ビルドとは：**
ソースコードを、実際にサーバー上で動作する形式に変換するプロセスです。Next.jsの場合、Reactコンポーネントを最適化されたJavaScriptファイルに変換したり、静的なHTMLを生成したりします。

**3-6: デプロイされたURLを確認する**

成功すると、以下の形式のURLが発行されます：
- `https://プロジェクト名.vercel.app`

このURLをクリックして、アプリが正常に動作するか確認してください。

### トラブルシューティング

**ビルドが失敗した場合：**

1. Vercelのダッシュボードで「Deployments」タブをクリック
2. 失敗したデプロイメントをクリック
3. 「Build Logs」を確認
4. エラーメッセージをGeminiに貼り付けて、解決策を聞いてみてください

**よくあるエラーと対処法：**

- `Environment variable not found`: 環境変数の設定を確認してください
- `Module not found`: 必要なパッケージがインストールされているか確認（`npm install`）
- `TypeScript error`: コードにエラーがないか確認（Antigravityで赤い波線がないか）

### この段階での提出物

- Vercelのデプロイ完了画面のスクリーンショット
- デプロイされたアプリのURL（`https://xxx.vercel.app`）

## Step 4: 動作確認とSupabaseの設定更新（所要時間：約15分）

### 手順

**4-1: PCで動作確認する**

デプロイされたURLにアクセスして、以下を確認してください：

- 画面が正しく表示される
- ログイン機能が動作する（Googleでログインできる）
- データの保存・取得ができる

**4-2: Supabaseの設定を更新する（Googleログインを使う場合）**

Googleログインでログインする場合、Supabaseの「Site URL」と「Redirect URLs」を更新する必要があります。

1. Supabaseダッシュボードを開く
2. 「Authentication」→「URL Configuration」をクリック
3. 「Site URL」を以下に変更：
   ```
   https://あなたのプロジェクト名.vercel.app
   ```
4. 「Redirect URLs」に以下を追加：
   ```
   https://あなたのプロジェクト名.vercel.app/**
   ```
5. 「Save」をクリック

これで、Google認証後にデプロイされたアプリにリダイレクトされるようになります。

**4-3: スマホで確認する**

1. デプロイされたURLをLINEやメールで自分のスマホに送る
2. スマホでURLを開く
3. WiFiを切って、4G/5Gで開いてみる
   - これにより、本当にインターネット上で動作していることを確認できます
4. ログインして、機能が正常に動作するか確認

### この段階での提出物

- スマホで開いたアプリのスクリーンショット（2枚以上）

## うまくいかないときは

### GitHubにプッシュできない

- ユーザー名とメールアドレスが設定されているか確認（`git config --list`）
- リモートURLが正しいか確認（`git remote -v`）
- 認証情報を求められたら、GitHubのパスワードまたはPersonal Access Tokenを入力

**Personal Access Token とは：**
GitHubのパスワードの代わりに使う認証トークンです。セキュリティのため、GitHubは通常のパスワードでのコマンドラインからの認証を推奨していません。GitHub Settings → Developer settings → Personal access tokens で作成できます。

### Vercelでビルドが失敗する

- ビルドログを確認して、エラーメッセージを特定
- エラーメッセージをGeminiに伝えて、解決策を聞く
- 環境変数が正しく設定されているか確認

### デプロイ後にアプリが動かない

- ブラウザの開発者ツール（F12）→ Consoleタブでエラーを確認
- 環境変数が正しいか確認
- Supabaseの「Site URL」と「Redirect URLs」が正しいか確認

### Googleログインが動かない

- Supabaseの「Authentication」→「URL Configuration」を確認
- Supabaseの「Providers」でGoogle認証が有効か確認
- Vercelの環境変数が正しいか確認
