## 宿題を始める前に：重要な概念の復習

### APIとは何か

API（Application Programming Interface）は、異なるソフトウェア同士が通信するための「窓口」です。

日常的な例えで説明すると、レストランに行ったとき、厨房に直接入って料理を作ることはできません。代わりに、メニュー（どんな料理が注文できるか）を見て、ウェイター（API）に注文を伝えます。ウェイターは厨房に注文を届け、できあがった料理を持ってきてくれます。

APIも同様に：
- **リクエスト**: 「この情報をください」という依頼
- **レスポンス**: サーバーから返ってくるデータ
- **エンドポイント**: リクエストを送る先のURL

という構造になっています。

### REST APIの基本

現在、最も一般的なAPIの形式が「REST API」です。

**REST（Representational State Transfer）とは：**
Webサービス間でデータをやり取りするための設計原則（アーキテクチャスタイル）です。以下のような特徴があります：

- HTTPメソッドで操作を区別する
  - **GET**: データを取得する
  - **POST**: データを作成する
  - **PUT/PATCH**: データを更新する
  - **DELETE**: データを削除する
- URLでリソース（データ）を特定する
- 状態を保持しない（ステートレス）

### JSONとは何か

APIでデータをやり取りする際、最もよく使われる形式が「JSON」です。

**JSON（JavaScript Object Notation）とは：**
データを構造化して表現するためのテキスト形式です。人間が読みやすく、プログラムでも扱いやすい形式になっています。

例：
```json
{
  "name": "田中太郎",
  "age": 25,
  "skills": ["JavaScript", "React", "Python"],
  "isStudent": false
}
```

構造：
- `{}` で囲まれた部分が「オブジェクト」
- `"キー": 値` の形式でデータを格納
- `[]` で囲まれた部分が「配列」（複数のデータのリスト）
- 値には文字列、数値、真偽値、配列、オブジェクトなどが使える

### APIキーとは何か

多くのAPIは、利用者を識別するために「APIキー」を要求します。

**APIキーとは：**
サービスを利用するための「パスワード」のようなものです。APIキーを使うことで：
- 誰がAPIを使っているかを特定できる
- 利用制限（1日1000回までなど）を設定できる
- 不正利用を防止できる

APIキーは絶対に公開してはいけません。GitHub等にコードをアップロードする際、APIキーがコードに含まれていると、悪意のある人に利用されてしまう可能性があります。

## 課題: 自分のアプリの「完璧な設計図」を作る

この宿題では、バイブコーディングの8ステップワークフローの前半部分（Step 1〜6）を実践します。設計図ができれば、実装はAIがやってくれます。

### バイブコーディング8ステップの復習

1. **アイデア**: 作りたいものを決める
2. **AIと雑談**: アイデアを膨らませる
3. **コンセプト固め**: 核となる価値を明確にする
4. **要件定義**: 誰のための、何を解決するアプリか
5. **Deep Research**: 技術的なリスクを調査
6. **タスク分解**: 実装計画を作成
7. **実装**: AIと一緒にコードを書く
8. **テスト・改善**: 動作確認と修正

## Step 1: アイデアを決める（所要時間：約10分）

### 手順

自分が「本当に作りたい」「自分で使いたい」と思えるアプリのアイデアを1つ決めてください。

アイデアを考えるヒント：

**日常の不便を解決する系**
- 家計簿アプリ（レシートをAIで読み取り）
- 献立提案アプリ（冷蔵庫の中身からレシピ提案）
- タスク管理アプリ（音声入力対応）
- 習慣トラッカー（毎日の行動を記録）

**趣味・学習系**
- 読書管理アプリ（感想をAIで要約）
- 英語学習アプリ（AIと英会話）
- 映画記録アプリ（おすすめをAIが提案）
- プログラミング学習アプリ

**クリエイティブ系**
- 日記アプリ（AIが質問してくれる）
- ブログエディタ（AIが文章を添削）
- アイデアメモアプリ（関連アイデアをAIが提案）

**ビジネス・仕事効率化系**
- 議事録作成アプリ（音声をテキスト化してAI要約）
- スケジュール管理アプリ
- 経費精算アプリ

もし何も思いつかない場合は、Geminiに相談してみてください：

```
私は初心者ですが、自分で使いたいと思えるWebアプリを作りたいです。
以下の条件でアイデアを10個出してください。
- 自分一人で使うもの
- シンプルな機能
- 毎日使いたくなるもの
- AIを活用できるもの
```

### この段階での提出物

決めたアイデアを1〜2行で記述してください。

例：「日々の支出を簡単に記録できる家計簿アプリ。カテゴリ分けをAIが自動でやってくれる。」

## Step 2-3: AIと雑談してコンセプトを固める（所要時間：約15分）

### 手順

Geminiと対話して、アイデアを具体化していきます。

**2-1: まずは雑談から**

以下のプロンプトをGeminiに送ってください（アイデアの部分は自分のものに置き換え）：

```
私は「〇〇アプリ」を作りたいと考えています。

このアプリについて、いくつか質問してください。
あなたの質問に答えることで、私のアイデアをより具体的にしたいです。

まずは3つ質問をお願いします。
```

Geminiからの質問に答えていくと、自分のアイデアが整理されていきます。

**2-2: コンセプトを言語化する**

雑談が終わったら、以下のプロンプトを送ってください：

```
これまでの会話を踏まえて、このアプリの「コンセプト」を以下の形式でまとめてください。

1. このアプリを一言で表すと？（キャッチコピー）
2. このアプリが解決する「たった1つの課題」は？
3. このアプリの「他にはない特徴」は？
4. 使った人が最初に感じる「感動ポイント」は？
```

Geminiの回答を読んで、しっくりこない部分があれば「〇〇の部分は違和感があります。もう少し考えてください」と伝えてみてください。

### この段階での提出物

Geminiとの会話のスクリーンショット、または会話内容をまとめたテキスト

## Step 4: 要件定義を作成する（所要時間：約20分）

### 要件定義とは何か

要件定義とは、「このアプリは何をするものか」を明確に文書化することです。実装を始める前に、以下を明確にしておく必要があります：

- **ターゲットユーザー**: 誰がこのアプリを使うのか
- **解決する課題**: ユーザーのどんな問題を解決するのか
- **MVP（Minimum Viable Product）**: 最小限、何ができれば「使えるアプリ」と言えるか

**MVPとは：**
「実用最小限の製品」という意味です。すべての機能を最初から作ろうとすると、永遠に完成しません。まずは「これだけあれば使える」という最小限の機能に絞って作り、後から機能を追加していくアプローチです。

### 手順

以下のプロンプトをGeminiに送ってください：

```
このアプリの要件定義を作成してください。

アプリ名: [あなたのアプリ名]
コンセプト: [Step 2-3で決めたコンセプト]

以下の項目を埋めてください。

## 1. ターゲットユーザー
- メインユーザー層（年齢、職業、特徴など）
- ユーザーのITリテラシーレベル
- 利用シーン（いつ、どこで使うか）

## 2. 解決する課題
- ユーザーが現在困っていること
- 既存の解決方法とその問題点
- このアプリが提供する解決策

## 3. 機能一覧
- 必須機能（MVPに含める）
- あったら嬉しい機能（後から追加）
- 将来的に検討する機能

## 4. 画面構成
- 必要な画面の一覧
- 各画面の主な役割

## 5. データ構造
- 保存するデータの種類
- データ間の関係

## 6. 技術要件
- 使用する技術スタック（フレームワーク、データベースなど）
- 外部API（使う場合）
- 特別な要件（リアルタイム更新、オフライン対応など）
```

Geminiが生成した要件定義を読んで、修正が必要な部分があれば指摘してください。

### この段階での提出物

完成した要件定義（Geminiが生成したもの + あなたの修正）

## Step 5: Deep Research（所要時間：約15分）

### Deep Researchとは何か

要件定義ができたら、その要件を実現するための「技術的な調査」を行います。具体的には：

- 使おうとしている技術が本当に要件を満たせるか
- 難しい部分はないか
- 代替案はないか

を調べます。

### 手順

以下のプロンプトをGeminiに送ってください（要件定義の内容を貼り付けて）：

```
以下の要件を、Next.js + Supabase + Tailwind CSS で実装しようとしています。
技術的なリスクと、代替案を教えてください。

[要件定義をここに貼り付け]

具体的に知りたいこと：
1. この要件で技術的に難しい部分はどこか
2. それぞれの難しい部分の解決策
3. 実装に必要なライブラリやAPI
4. 初心者が特に注意すべきポイント
5. 想定される実装時間（各機能ごと）
```

Geminiの回答を読んで、不明な点があれば追加で質問してください。

### この段階での提出物

リスクと対策のリスト（Geminiの回答をまとめたもの）

---

## Step 6: タスク分解（所要時間：約20分）

### タスク分解とは何か

要件定義と技術調査ができたら、実際の「作業リスト」に落とし込みます。大きな目標を、小さな作業に分解することで、何から始めればいいかが明確になります。

### 手順

Antigravityを開いて、AIエージェントに以下のプロンプトを送ってください：

```
以下の要件定義に基づいて、実装タスクを分解してください。

[要件定義をここに貼り付け]

以下の形式でお願いします：

## フェーズ1: 環境構築（所要時間: 〇〇分）
- [ ] タスク1-1: 〇〇〇〇
- [ ] タスク1-2: 〇〇〇〇
...

## フェーズ2: 基本機能の実装（所要時間: 〇〇分）
- [ ] タスク2-1: 〇〇〇〇
- [ ] タスク2-2: 〇〇〇〇
...

## フェーズ3: 認証機能（所要時間: 〇〇分）
...

## フェーズ4: データベース連携（所要時間: 〇〇分）
...

## フェーズ5: UI/UXの改善（所要時間: 〇〇分）
...

## フェーズ6: テストとデプロイ（所要時間: 〇〇分）
...

各タスクは、1回のAIへの指示で完了できる粒度にしてください。
チェックボックス形式にして、進捗を管理できるようにしてください。
```

AIが生成したタスクリストを確認して、以下をチェックしてください：

- 各タスクが具体的で、次に何をすればいいかわかるか
- 順番は適切か（依存関係のあるタスクは後に来ているか）
- 抜け漏れはないか

問題があれば、「タスク〇〇はもう少し細かくしてください」「〇〇の機能のタスクが抜けています」と指示してください。

### この段階での提出物

完成したタスクリスト（Markdown形式）

## （オプション）Step 7: 実装を始める（所要時間：30分以上）

時間がある方は、タスクリストの最初の数個を実際に実装してみてください。

### 手順

1. Antigravityで新しいプロジェクトを作成（Next.jsテンプレート）
2. タスクリストの「フェーズ1」の最初のタスクをAIエージェントに依頼
3. AIがコードを生成したら、プレビューで確認
4. 問題なければ「次のタスクをお願いします」と伝える
5. 問題があれば「〇〇の部分を修正してください」と伝える

この繰り返しで、アプリが少しずつ形になっていきます。

### この段階での提出物（オプション）

- 実装した画面のスクリーンショット
- または、進捗状況のメモ

## うまくいかないときは

### アイデアが浮かばない

- 「毎日使うアプリ」を思い浮かべて、不満点を探してみてください
- 「こういうアプリがあったらいいのに」と思ったことを思い出してください
- Geminiに「私の趣味は〇〇です。関連するアプリのアイデアを出して」と聞いてみてください

### 要件定義が難しい

- 最初から完璧を目指さなくて大丈夫です
- 「誰の」「何の課題を」「どう解決するか」の3点だけ明確にできればOKです
- わからない部分は空欄のまま進んで、後から埋めても大丈夫です

### タスク分解が細かすぎる/粗すぎる

- 「1回のAIへの指示で完了できる粒度」を意識してください
- 粗すぎる場合：「このタスクをもっと細かく分解してください」
- 細かすぎる場合：「このタスクをもう少しまとめてください」

### 技術的な部分がわからない

- すべてを理解する必要はありません
- AIに「〇〇とは何ですか？初心者にもわかるように説明してください」と聞いてください
- わからない部分はそのままにして、実装時にAIに任せても大丈夫です
