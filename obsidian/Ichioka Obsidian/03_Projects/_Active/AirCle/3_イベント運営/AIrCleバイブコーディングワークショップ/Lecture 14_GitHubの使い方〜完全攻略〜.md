## 【Part 2: Gitの概念（セーブポイント）】

### スライド 4: Gitの3段階

**タイトル**: セーブまでの道のり **ビジュアル**: **内容**:

1. **作業場（Working Tree）**: いま書いている場所。散らかっている。
    
2. **カゴに入れる（Staging）**: 「これを保存するぞ」と選んだもの置き場。
    
3. **セーブする（Commit）**: 「メッセージ」をつけて保存確定。倉庫に格納。 **発表者ノート**: いきなりセーブされるわけじゃありません。「カゴに入れて（Add）」から「レジを通す（Commit）」という2段階があることを覚えておいてください。
    

### スライド 5: コミット（Commit）とは？

**タイトル**: 歴史を刻む **内容**:

- ゲームのセーブデータ作成と同じ。
    
- ただし、必ず**「コミットメッセージ」**（メモ）が必要。
    
- 例：「ログイン機能追加」「バグ修正」「ボタンを赤くした」
    
- これを積み重ねることで、開発の歴史（ログ）が作られる。 **発表者ノート**: 「あとで自分が読んだ時にわかるメモ」を残すのがマナーです。「あ」とか適当に書くと未来の自分が困ります。

## 【Part 3: Antigravity（GUI）での操作】

### スライド 6: 黒い画面は使わない

**タイトル**: マウスだけでOK **内容**:

- 昔のエンジニアは `git add .` `git commit...` とコマンドを打っていた。
    
- 今は **Antigravity（VS Code）の「ソース管理」タブ** で全部できる。
    
- 左側のメニューにある「枝（ブランチ）」のようなマークをクリック。 **発表者ノート**: 便利な時代になりました。難しいコマンドは覚えなくていいです。ポチポチするだけで終わります。
    

### スライド 7: 【演習】最初のコミット

**タイトル**: 🛠️ 初セーブ **手順**:

1. 枝マークのタブを開く。
    
2. 「変更」欄にファイルがいっぱいあるはず。
    
3. **「＋」ボタン**（変更をステージング）を押す ➡ カゴに入った状態。
    
4. メッセージ欄に **"First Commit"** と書く。
    
5. **「コミット」ボタン** を押す。 **発表者ノート**: これでリストが消えましたね？ それは「全ての変更が記録された」からです。おめでとうございます、最初のセーブポイントが作られました。

## 【Part 4: GitHubへのアップロード】

### スライド 8: GitHubとは？

**タイトル**: インターネット上の巨大倉庫 **内容**:

- **Git**: あなたのPC内にある記録（ローカル）。
    
- **GitHub**: ネット上にある記録置き場（リモート）。
    
- PCが壊れても、GitHubにあれば復元できる。
    
- 世界中のエンジニアとコードを共有できる。 **発表者ノート**: GitとGitHubは別物です。「自分の日記帳（Git）」と「ブログ（GitHub）」くらい違います。
    

### スライド 9: 【演習】倉庫の契約（リポジトリ作成）

**タイトル**: 🛠️ 置き場を作る **手順**:

1. ブラウザで `github.com` にログイン。
    
2. 右上の **「+」** -> **「New repository」**。
    
3. Repository name: `my-vibe-app`
    
4. Public（公開）か Private（自分だけ）を選ぶ（練習ならPublicでOK）。
    
5. **「Create repository」**。 **発表者ノート**: これでネット上に空っぽの箱ができました。ここにPCの中身を転送します。
    

### スライド 10: 【演習】紐付けとプッシュ（Push）

**タイトル**: 🛠️ 荷物を送る **手順**:

1. GitHubの画面にコマンドが表示されているはず。
    
2. `…or push an existing repository from the command line` の下の3行をコピー。
    
3. Antigravityのターミナルを開いて貼り付け、Enter！
    
    - `git remote add...`（宛先登録）
        
    - `git push...`（送信！） **発表者ノート**: ここだけはコマンドを使ったほうが確実です。コピペでOK。次回からはボタン一つで送れるようになります。
        

### スライド 11: 確認

**タイトル**: 🛠️ 届いたかな？ **内容**:

- GitHubの画面をリロード（再読み込み）する。
    
- さっきまでの空っぽの画面が消えて、ファイル一覧が表示されたら成功！
    
- **`App.jsx`** などの中身が見れるか確認しよう。 **発表者ノート**: これであなたのコードはクラウドに保存されました。もうPCを窓から投げ捨てても大丈夫です（投げないでください）。
    

### スライド 12: 【超重要】環境変数の罠

**タイトル**: ⚠️ 鍵は送らない **内容**:

- GitHubを見てみよう。**`.env` ファイルがありませんね？**
    
- これは `.gitignore` という設定ファイルが「鍵は送るな！」と止めてくれているから。
    
- **絶対にAPIキーをGitHubに上げてはいけません。**（数秒で盗まれて悪用されます） **発表者ノート**: Antigravity賢いですね。危険物は自動で検知して、倉庫への配送をブロックしてくれています。

## 【Part 5: パラレルワールド（ブランチ）】

### スライド 13: 開発の鉄則

**タイトル**: 本番環境を汚さない **ビジュアル**: **内容**:

- 動いている「メインのコード（main）」を直接いじって、バグらせたら大変。
    
- **「ブランチ（Branch）」**：メインから枝分かれした「コピー世界」を作る。
    
- コピー世界でいくら失敗しても、メイン世界には影響しない。 **発表者ノート**: プロは「main」を直接触りません。必ず「実験室」を作ってそこで作業します。
    

### スライド 14: 【演習】ブランチを切る

**タイトル**: 🛠️ 世界線を変える **手順**:

1. Antigravityの左下にある **`main`** という文字をクリック。
    
2. **`Create new branch...`** を選択。
    
3. 名前：`feature/new-design` （新しいデザイン機能、の意味）。
    
4. 左下の文字が `feature/new-design` に変わったのを確認。 **発表者ノート**: 今、あなたは「メイン世界」とは別の次元にいます。ここで何をしても、元のコードは壊れません。
    

### スライド 15: 実験と合体（Merge）

**タイトル**: 🛠️ 統合する **内容**:

- 実験用ブランチでコードを書き換え、コミットする。
    
- うまくいったら、メイン世界に戻る（Checkout main）。
    
- **「マージ（Merge）」**: 実験世界の変更を、メイン世界に取り込む。
    
- これで安全に機能追加完了！ **発表者ノート**: 個人開発ならmain直書きでもいいですが、「ブランチを切る」癖をつけておくと、カッコいいし安全です。

## 【Part 6: AIとGit】

### スライド 16: コミットメッセージもAIに

**タイトル**: 面倒な入力を丸投げ **内容**:

- 「変更内容の要約を書くの面倒くさい...」
    
- Antigravity（Source Controlパネル）の入力欄の近くにある **「キラキラボタン（✨）」** を押す。
    
- **AIが変更内容（Diff）を見て、「ボタンの色を修正し、バグを削除」みたいなメッセージを勝手に書いてくれる！** **発表者ノート**: 地味に一番便利な機能です。英語で書かれることもありますが、それはそれでプロっぽいです。
    

### スライド 17: コンフリクトの解消

**タイトル**: 衝突しても怖くない **内容**:

- 「マージしようとしたらファイルが衝突した！（Conflict）」
    
- 初心者最大のパニックポイント。
    
- Antigravityのエージェントに聞く：**「コンフリクトしてる。現在の変更（Current）を優先して解消して」**
    
- AIがコードを整理して直してくれる。 **発表者ノート**: 赤いビックリマークが出ても慌てないで。AIという優秀な仲裁人がいます。

## 【Part 7: まとめ】

### スライド 18: バイブコーディング的運用

**タイトル**: ルールは3つだけ **内容**:

1. 作業を始める前に **ブランチ** を作る（推奨）。
    
2. 一区切りついたらこまめに **コミット** する（セーブ）。
    
3. 作業が終わったら **プッシュ** する（倉庫へ）。 **発表者ノート**: これだけ守れば、データが消える悲劇は起きません。
