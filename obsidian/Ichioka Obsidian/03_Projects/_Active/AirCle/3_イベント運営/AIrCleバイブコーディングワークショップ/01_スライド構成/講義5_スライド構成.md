## 【Part 2: React/Next.jsの理論（スライド4-9）】

### スライド4: React = 「レゴブロック」
**タイトル**: コンポーネント（Component）  
**内容**:
- Webページを「一枚の絵」として描くのではなく、**「部品（ブロック）」**を組み合わせて作る。
- ヘッダー部品、ボタン部品、カード部品...
- これらを `Components` フォルダに入れておき、パズルのように組み合わせるのがReactです。  
**発表者ノート**: 「Button.tsx」というファイルを作りましたよね？あれは「ボタンというブロック」を定義していたんです。

### スライド5: State（ステート）= 「記憶」
**タイトル**: ただの変数とは違う  
**内容**:
- **普通の変数**: 書き換えても、画面は変わらない（リロードが必要）。
- **State (`useState`)**: **書き換えた瞬間、画面が自動で再描画される魔法の変数。**
- 「おみくじ結果」や「入力フォームの文字」は、全部これ。  
**発表者ノート**: `useState`。これは「画面を書き換えるスイッチ付きの箱」です。

### スライド6: Server vs Client
**タイトル**: 厨房と客席  
**内容**:
- **Server Component (厨房)**: サーバー側。DBに近い。安全。でも**客の操作（クリック）はできない**。
- **Client Component (客席)**: ブラウザ側。**クリックや入力ができる**。でもDBに直接触るのは危険。
- **`"use client"`** = 「これは厨房じゃなくて、客席（ブラウザ）で動かす部品ですよ」という宣言。  
**発表者ノート**: ここが今日一番の山場です。Next.jsには「2つの世界」があります。

### スライド7: 使い分けの基準
**タイトル**: どっちで書く？  
**内容**:
- Q. **データベースやAPIキーを使いますか？** → YES ➡ **Server Component** (デフォルト)
- Q. **`onClick` や `useState` を使いますか？** → YES ➡ **Client Component** (`"use client"` をつける)  
**発表者ノート**: 迷ったらこれを見てください。

## 【Part 3: GitHubでバックアップ（スライド10-15）】

### スライド10: Git = 「セーブポイント」
**タイトル**: ゲームの「セーブポイント」  
**内容**:
- ファイルを上書き保存すると、過去の状態は消える。
- **Git** は、過去のあらゆる時点に「セーブポイント（コミット）」を作れる。
- 失敗したら、昨日のセーブポイントまで時間を巻き戻せる。  
**発表者ノート**: 「バグった！終わった！」とならないためのタイムマシン。それがGitです。

### スライド11: GitHub = 「コードの図書館」
**タイトル**: コードの「図書館 兼 SNS」  
**内容**:
- Gitは「自分のPC内」の履歴。
- **GitHub** は、それを「クラウド（ネット上）」に保存する場所。
- 全世界のエンジニア（やAI）がコードを公開・共有している。
- **デプロイ（Vercel）** も、ここからコードを引っ張っていく。  
**発表者ノート**: GitHubは「エンジニアのインスタグラム」です。

### スライド12: 【演習】最初のコミット
**タイトル**: 🛠️ 初セーブ  
**手順**:
1. Antigravityの左メニュー「ソース管理」タブを開く。
2. 「変更」欄にファイルがいっぱいあるはず。
3. **「＋」ボタン**（変更をステージング）を押す ➡ カゴに入った状態。
4. メッセージ欄に **"First Commit"** と書く。
5. **「コミット」ボタン** を押す。  
**発表者ノート**: これでリストが消えましたね？ それは「全ての変更が記録された」からです。

### スライド13: 【演習】GitHubにPush
**タイトル**: 🛠️ 倉庫に送る  
**手順**:
1. `github.com` にログイン。
2. 右上の **「+」** -> **「New repository」**。
3. Repository name: `my-vibe-app`
4. Public（公開）か Private（自分だけ）を選ぶ。
5. **「Create repository」**。
6. GitHubの画面に表示されるコマンドをコピーして、Antigravityのターミナルで実行。  
**発表者ノート**: これであなたのコードはクラウドに保存されました。

### スライド14: 【超重要】環境変数の罠
**タイトル**: ⚠️ 鍵は送らない  
**内容**:
- GitHubを見てみよう。**`.env` ファイルがありませんね？**
- これは `.gitignore` という設定ファイルが「鍵は送るな！」と止めてくれているから。
- **絶対にAPIキーをGitHubに上げてはいけません。**（数秒で盗まれて悪用されます）  
**発表者ノート**: Antigravity賢いですね。危険物は自動で検知して、倉庫への配送をブロックしてくれています。

## 【Part 4: Vercelでデプロイ（スライド15-19）】

### スライド15: Vercelとは？
**タイトル**: Next.jsの生みの親  
**内容**:
- Next.jsを開発している会社が作った、Web公開サービス。
- **特徴**:
  - **設定ゼロ**：Next.jsなら自動で認識して最適化してくれる。
  - **爆速**：世界中にサーバーがある。
  - **無料**：個人開発なら十分すぎる無料枠。  
**発表者ノート**: Next.jsで作ったなら、Vercel一択です。

### スライド16: 【演習】Vercelに接続
**タイトル**: 🛠️ 店を構える  
**手順**:
1. `vercel.com` にアクセス -> **「Sign Up」**。
2. **「Continue with GitHub」** を選ぶ（これが重要！）。
3. ログインすると、ダッシュボードが表示される。
4. **「Add New...」** -> **「Project」**。
5. **「Import Git Repository」** のリストに、前回作った `my-vibe-app` があるはず。
6. **「Import」** ボタンを押す。  
**発表者ノート**: 倉庫にある荷物を選んで「これを店に出すよ」と指示します。

### スライド17: 【最重要】環境変数の設定
**タイトル**: ⚠️ 鍵を忘れていませんか？  
**内容**:
- ここで「Deploy」を押すと、**エラーになります（または動きません）。**
- なぜ？ -> **「鍵（.env）」がGitHubにないから。**
- **店舗（Vercel）の金庫に、直接鍵を入れる必要があります。**
- Import画面にある **「Environment Variables」** をクリックして、SupabaseやGeminiの鍵を登録する。  
**発表者ノート**: 初心者が一番ハマる罠です。「ローカルでは動いたのに！」の原因は99%これです。

### スライド18: デプロイ実行！
**タイトル**: 🛠️ スイッチオン  
**手順**:
1. 鍵の設定が終わったら...
2. **「Deploy」** ボタンをクリック！
3. 画面が切り替わり、ビルド（建築）が始まる。
4. 1〜2分待つ...
5. **「Congratulations!」** の文字が出たら成功！
6. **`https://my-vibe-app.vercel.app`** のようなURLでアプリが開く！  
**発表者ノート**: おめでとうございます！ これがあなたのアプリの「世界共通住所」です。

### スライド19: 実機テスト
**タイトル**: 🛠️ 手のひらの上の魔法  
**手順**:
1. そのURLをスマホに送る（LINEやAirDropで）。
2. スマホで開く（WiFiを切って4G/5Gで）。
3. ログインして、タスクを追加してみる。
4. **PCで見ると、リアルタイムで同期されている！**  
**発表者ノート**: 「自分のPC」という殻を破り、インターネット上で動いています。
