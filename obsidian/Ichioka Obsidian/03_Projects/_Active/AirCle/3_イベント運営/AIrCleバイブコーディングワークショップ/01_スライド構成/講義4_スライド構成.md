## 【Part 1: イントロダクション（スライド1-3）】

### スライド1: タイトル
**タイトル**: 講義4: API連携とAI統合  
**サブタイトル**: 世界中のデータを取り込み、アプリに「知能」を宿す  
**ビジュアル**: 自分のアプリに世界中からデータが流れ込んでくる図  
**発表者ノート**: ようこそ。前回、データベースと認証を学びました。今日は「外の世界」と繋がり、アプリに「頭脳」を組み込みます。

### スライド2: 今日のゴール
**タイトル**: 今日できるようになること  
**内容**:
1. 🔌 **API** の仕組みを理解し、天気データを取り込む
2. 🧠 **Gemini API** を組み込み、アプリにAI機能を追加する
3. 🗺️ **バイブコーディングの全ワークフロー**をマスターする  
**発表者ノート**: 今日の目標は「世界中のデータを使えるアプリ」と「AIがアドバイスしてくれるアプリ」を作ることです。

### スライド3: APIって何？
**タイトル**: レストランで例えると？  
**内容**:
- **客（あなたのアプリ）**: 「ハンバーグ（天気データ）ください」
- **厨房（サーバー）**: 実際にデータを作っている場所。客は立ち入り禁止。
- **ウェイター（API）**: 客の注文を聞いて厨房に伝え、料理を持って帰ってくる人。
- **APIがないと、厨房に入れないのでデータがもらえない。**  
**発表者ノート**: APIは「注文の窓口」であり「運び屋」です。

## 【Part 2: 外部APIを使う（スライド4-10）】

### スライド4: Open-Meteo（天気API）
**タイトル**: 登録不要の神API  
**内容**:
- 世の中のAPIは「登録（API Key）」が必要なものが多い（面倒くさい）。
- **Open-Meteo** は、登録不要・無料で天気データをくれる神サービス。
- 練習にはこれが最適！  
**発表者ノート**: Google Mapsなどは登録が必要ですが、今回はハードルを下げるために、登録なしで使えるこれを使います。

### スライド5: 【演習】ブラウザで実験
**タイトル**: 🛠️ 直叩きしてみる  
**手順**:
1. ブラウザの新しいタブを開く。
2. アドレスバーに `https://api.open-meteo.com/v1/forecast?latitude=35.689&longitude=139.691&current_weather=true` と入力してEnter。
3. 画面に文字（JSON）がブワッと出たら成功！  
**発表者ノート**: 見えましたか？ これが「生のデータ」です。東京の天気が返ってきました。

### スライド6: Antigravityで実装
**タイトル**: 🛠️ 部品の発注  
**指示（Antigravity）**: **「`Weather.jsx` という新しいコンポーネントを作って。 Open-MeteoのAPIを使って、東京の現在の天気と気温を取得して表示して。 デザインはシンプルに、晴れなら太陽、雨なら傘のアイコンを出して。」**  
**発表者ノート**: 「fetchを使って...」とか言わなくていいです。「Open-Meteoを使って」と言えば、AIはURLも使い方も全部知っています。

### スライド7: データ連携の真髄
**タイトル**: ただ表示するだけじゃつまらない  
**内容**:
- 天気を表示するだけなら、天気予報アプリを見ればいい。
- **「天気によって、Todoアプリの挙動を変える」** のが面白い。
- 例：雨なら「傘を持っていく」というタスクを自動追加する。  
**発表者ノート**: ここからがプログラミングの醍醐味です。「外部データ」をトリガーにして、アプリを賢くします。

### スライド8: API Keyの管理
**タイトル**: 鍵は玄関マットの下に置くな  
**内容**:
- API Keyをコードに直接書くと、GitHubで公開された時に**全世界に流出**する。
- **`.env` ファイル** に書いて、Gitにはアップしないのが鉄則。
- AIに聞こう：**「API Keyを安全に管理する方法を教えて」**  
**発表者ノート**: 「パケ死」ならぬ「クラウド破産」を防ぐために、Keyの管理だけは慎重に。

## 【Part 3: AI統合（スライド9-14）】

### スライド9: Gemini APIとは？
**タイトル**: アプリに「脳」を移植する  
**内容**:
- 天気APIは「事実を返すだけ」。
- **Gemini API** は「考えて、生成する」。
- 違い：決まった答えがない。文脈を理解する。  
**発表者ノート**: 天気APIは「新聞配達」ですが、Gemini APIは「コンサルタント」です。

### スライド10: Google AI Studio
**タイトル**: 脳みその調達場所  
**手順**:
1. `aistudio.google.com` にアクセス（Googleアカウントでログイン）。
2. **「Get API key」** をクリック。
3. **「Create API key」** でキーを発行し、コピーする。  
**発表者ノート**: これが「知能へのアクセス権」です。絶対に他人に教えてはいけません。

### スライド11: AI機能の実装
**タイトル**: 🛠️ 相談窓口を作る  
**指示**: **「タスクリストの下に『🔥 修造に相談する』というボタンを作って。 押すと、現在のタスク一覧を読み取って、Geminiに送信する関数も作って。 プロンプトは『あなたは熱血コーチです。以下のタスクリストを見て、ユーザーを全力で励まし、具体的なアドバイスを1つください』として。」**  
**発表者ノート**: モデルは「Flash」を使うよう指示してください。軽くて速くて無料枠が広いからです。

### スライド12: JSONモード（発展）
**タイトル**: AIに枠をはめる  
**内容**:
- 励ましも嬉しいけど、**「タスクを分解」** してくれたらもっと便利。
- 例：「カレーを作る」→ 「肉を買う」「野菜を切る」「煮込む」に分解して、**自動でリストに追加**してほしい。
- プロンプトの最後に： **「回答は必ずJSON形式にしてください。例：`[{"task": "野菜を切る"}, ...]`」**  
**発表者ノート**: AIの答えを「読む」だけでなく「機能として使う」ための魔法です。

## 【Part 4: バイブコーディングの全ワークフロー（スライド13-18）】

### スライド13: 勝利の8ステップ
**タイトル**: The Vibe Coding Flow  
**内容**:
1. **Start**: エディタ（Antigravity）を開く
2. **Base**: Next.jsで土台作成（UI・ルーティング）
3. **DB**: Supabaseで認証・CRUD確保
4. **Plan**: 要件定義（誰の・何の課題？）
5. **Research**: 実現可能性の調査
6. **Ticket**: **タスク分解プロンプト**（ここが重要！）
7. **Build**: チケット消化（実装と確認の反復）
8. **Finish**: テスト・デプロイ準備  
**発表者ノート**: この流れを体に叩き込んでください。これが「作れる人」の脳内です。

### スライド14: タスク分解プロンプト
**タイトル**: ★今日の最重要テクニック  
**プロンプト例**:
```
決定した要件を元に、実装ステップを「細かいタスクリスト」に分解してください。
1つのタスクは「1ファイル修正」または「1機能実装」レベルまで小さくすること。
【フォーマット】
- [ ] Task 1: 〇〇画面のUI作成
- [ ] Task 2: Supabaseへの接続設定
- [ ] Task 3: ...
このリストを常に記憶し、私が「次」と言ったら、次のタスクを実行してください。
```  
**発表者ノート**: これをAIに読み込ませます。すると、AIが優秀なプロジェクトマネージャーに変身します。

### スライド15: 実装ループ
**タイトル**: ひたすら消化する  
**内容**:
1. AI：「次は Task 1: UI作成 です。実行しますか？」
2. あなた：「やって」
3. AIがコードを書く → プレビュー確認。
4. あなた：「OK、次」または「ここ直して」
5. **リストの `[ ]` が `[x]` に変わっていく快感！**  
**発表者ノート**: あなたは「承認ボタン」を押す係です。迷わず、ズレ修正だけに集中できます。

### スライド16: 参照（Context）の重要性
**タイトル**: 記憶を保たせる  
**内容**:
- AIは前の話を忘れがち。
- 毎回、**「Step 4の要件定義」と「Step 6のタスクリスト」を参照（メンション）** させながら指示する。
- 「今の実装、要件とズレてない？」と定期的にチェックさせる。  
**発表者ノート**: Antigravityの `@Files` 機能を使いましょう。「計画書」を常に机の上に置いておくイメージです。
